# 模块 01：战斗模式、队伍构筑与信息可见性

本文件定义“打什么、怎么开局、双方能看到什么”。

## 1. 战斗模式

|项|规则|
|---|---|
|战斗形式|单打 1v1|
|队伍规模|每队 3 名单位|
|等级口径|战斗内固定按 `Lv = 50` 计算|
|首发数量|每方 1 名|
|每回合选指令限时|每方 `30s`（`BattleFormatConfig.selection_deadline_ms = 30000`）|
|回合上限|`max_turn = 40`|
|触发链深上限|`BattleFormatConfig.max_chain_depth = 32`|
|投降|提供；触发后立即判该方负、对方胜|

## 2. 队伍构筑

|项|规则|
|---|---|
|同种单位|当前不限制，可重复|
|被动持有物|每单位最多 1 个；同队不可重复|
|主动道具|标准模式不启用|
|技能槽|常规技能 3 个|
|赛前配招|允许在战斗开始前按槽位覆盖本场 3 个常规技能；当前只支持常规技能，不支持奥义/被动/持有物覆盖|
|被动槽|1 个，不占主动技能槽|
|奥义槽|1 个，可用；其 `priority` 只能是 `+5` 或 `-5`|

补充规则：

1. `UnitDefinition.skill_ids` 固定表示默认装配的 3 个常规技能。
2. `UnitDefinition.candidate_skill_ids` 表示允许赛前换入的常规技能候选池；为空时，表示没有额外候选池。
3. `SideSetup.regular_skill_loadout_overrides` 的键固定为队伍槽位下标，不用 `unit_definition_id`，因为同队允许重复单位。

## 3. 当前唯一可见性模式

当前只保留 `prototype_full_open`。

|项|规则|
|---|---|
|战前公开信息|双方公开完整队伍战斗信息：单位身份、等级、本场已装备的常规技能列表、奥义、被动、被动持有物、基础战斗数值、稳定公开编号|
|首发选择方式|战斗开始前，各方从自己的 3 名单位中选择 1 名首发|
|是否盲选|否；首发选择前可见敌方完整队伍信息|
|战中公开|双方全部单位当前 HP、MP、能力阶段、当前 field、已公开单位身份、已生效被动持有物效果、以及仍挂载中的公开持续效果都公开|
|私有信息|随机值、未来抽样结果、内部调试字段不公开|

补充规则：

1. 规则层 full open，不等于 UI 层必须同时把所有细节塞到主界面。
2. 默认展示应优先给玩家看：当前出战单位、HP/MP、可选技能、当前 field、最关键的结算摘要。
3. 详细数值、持有物全文、复杂效果说明可以放在展开面板。
4. 当前 `prototype_full_open` 下，bench 单位的当前 HP/MP 与保留中的战斗态同样属于公开信息，不因离场变回私有信息。
5. 当前公开层只公开“这场战斗实际已装备的技能”，不公开候选技能池全集。

## 4. 重复单位的公开编号

|场景|规则|
|---|---|
|同队存在重复单位|必须给每个单位分配稳定公开编号，如 `P1-A / P1-B / P1-C`|
|用途|用于队伍预览、行动日志、回放、击倒记录|
|稳定性|同一场对局内编号不可变化|

## 5. 战斗开始初始化顺序

|步骤|规则|
|---|---|
|1|加载双方队伍、被动、被动持有物配置|
|2|公开双方完整队伍战斗信息并分配稳定公开编号|
|3|双方选择首发|
|4|双方首发同时入场|
|5|写入 `system:battle_header`：`visibility_mode / prebattle_public_teams / initial_active_public_ids_by_side / initial_field`（无 field 写 `null`）|
|6|先收集并结算当前 active 产生的 `on_enter`；统一效果排序只在本次 `on_enter` 批次内生效，不与其他触发点混排|
|7|若 `on_enter` 批次导致击倒，立即处理击倒窗口与强制补位，直到 active 稳定|
|8|以上述稳定战场为输入，统一检查一次 `battle_init` 触发的战斗开始时被动 / 持有物 / field 效果；统一效果排序只在本次 `battle_init` 批次内生效|
|9|若 `battle_init` 批次导致击倒，立即进入击倒窗口与强制补位，直到 active 再次稳定|
|10|若 `battle_init` 后稳定对位相较进入该批次前发生变化，则补跑一次 `on_matchup_changed` 批次|
|11|进入第 1 回合|

补充规则：

1. `battle_init` 是“战斗开始时统一检查一次”的批次，不因为后续有单位再次入场而重复触发。
2. 若首发 `on_enter` 期间发生击倒并完成强制补位，则 `battle_init` 读取补位后稳定下来的当前战场，不读取已离场单位。
3. `system:battle_header` 必须先于首条 `state:enter` 写入，且日志头中禁止泄露 `unit_instance_id` 等私有运行态 ID。
4. 若 `battle_init` 批次本身导致补位并形成新的稳定对位，进入第 1 回合前必须再执行一次 `on_matchup_changed`；该补跑批次只读取 `battle_init` 后稳定下来的当前战场，不会重放 `battle_init`。

## 6. 胜负与结束条件

|项|规则|
|---|---|
|通常胜利|对方无可上场单位|
|通常失败|己方无可上场单位|
|同窗双空|同一结算窗口双方都无可上场单位，则判平局|
|投降|触发即结束；该方负，对方胜|
|回合上限结束|第 40 回合完整结算后执行超时比较|

### 6.1 超时比较

比较顺序固定如下：

1. 先比剩余可上场单位数。
2. 再比全队当前 HP / 全队最大 HP 的占比，使用交叉相乘整数比较，禁止直接比浮点。
3. 再比全队当前 HP 总和。
4. 仍完全相同则记为平局。

补充规则：

- 超时比较固定按全队 3 名单位的 `max_hp` 总和作为分母；已倒下单位按 `current_hp = 0` 计入。

## 7. 玩家上手约束

|项|规则|
|---|---|
|技能说明|每个技能都必须有一句人话描述，不能只给公式|
|不可选原因|UI 必须提前告诉玩家“MP 不足”“当前没有可换单位”等原因|
|命中失败说明|若未命中，日志和 UI 要明确写成“未命中”，不要只显示“没有效果”|
|field 说明|当前 field 名称、剩余回合、核心效果要能一眼看到|
