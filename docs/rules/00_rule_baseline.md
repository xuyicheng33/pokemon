# 战斗系统规则总则（v0.6.3）

本文件定义当前战斗原型的全局规则口径。它不代替模块细则，而是负责把全局不变量和分叉原则写死。

## 1. 范围与目标

|项|规则|
|---|---|
|项目阶段|概念与原型期|
|当前范围|仅战斗系统，不包含探索、捕获、养成外流程|
|当前目标|先做出规则自洽、可回放、可回归，且形成“参考官方对战思路的项目化改写”1v1 回合制战斗闭环|
|体验定位|采用“完整公开队伍战斗信息”的项目化赛事式信息结构；强调参考官方思路，不做官方赛事规则逐条复刻|
|设计优先级|规则自洽 > 可玩闭环 > 扩展接口 > 数值平衡|
|实现原则|开发阶段默认 fail-fast，不做静默兜底，不做隐式兼容|

## 2. 权威文档边界

|项|规则|
|---|---|
|总则职责|定义术语、统一原则、官方参考口径、模块边界、主动分叉|
|模块职责|给出可直接编码的详细细则|
|记录文档职责|只记录决策背景与任务过程，不定义当前生效规则|
|冲突处理|总则摘要与模块细则冲突时，以模块细则为准；模块间冲突视为文档缺陷，需先修文档再开发|

## 3. 官方参考与主动分叉

### 3.1 主要官方参考

当前规则主要参考以下官方资料：

- Pokémon Brilliant Diamond and Pokémon Shining Pearl Trainers Guide: Battling  
  https://diamondpearl.pokemon.com/en-us/trainersguide/fundamentals/battling/
- Play! Pokémon Video Game Rules, Formats, and Penalty Guidelines  
  https://play.pokemon.com/en-gb/resources/documents/play-pokemon-vg-rules-formats/
- Pokémon Strategy: The Tapu Transform Terrains  
  https://www.pokemon.com/us/strategy/the-tapu-transform-terrains

### 3.2 当前明确对齐官方的部分

|模块|当前口径|
|---|---|
|基础对战骨架|速度决定先后，行动按顺序逐个结算|
|信息结构|参考官方 Team Preview / Team List 思路，在开战前就公开完整队伍战斗信息|
|单体目标|默认锁定敌方在场位置；位置上的旧单位离场后，新单位补位则成为新目标|
|换人与离场|离场会清除大部分临时战斗态；全局效果与队伍效果独立存在|
|状态语义|中毒、灼伤、麻痹、睡眠、畏缩的核心语义尽量贴近官方主系列|
|持有物思路|每单位最多 1 个被动持有物；同队不可重复|
|超时比较思路|用固定回合上限防止无限对局，比较顺序为先比存活单位，再比血量占比，再比总血量|
|限时思路|参考官方赛事时间控制思路，加入每回合选指令限时，防止拖时|

### 3.3 当前明确偏离官方的部分

这些是主动分叉，不是“以后发现写错了再改”的内容：

|分叉点|当前规则|
|---|---|
|资源系统|采用单位内共享 `MP`（每单位独立 MP 池），不采用官方每招式独立 `PP`|
|战斗规模|当前原型为 1v1、每队 3 单位，不直接照搬官方 VGC 双打|
|构筑限制|当前不限制同种单位，不使用官方“同国家图鉴号不可重复”限制|
|战前信息|采用完整公开队伍信息：公开敌方队伍名单、技能、被动、持有物与战斗数值；不逐字段照搬官方 Open Team List 文书格式|
|道具系统|标准战斗不启用主动道具，只保留被动持有物；若未来要做剧情/PVE 模式，再单独扩展|
|命中体系|采用独立闪避率，不采用官方命中/闪避阶段制|
|状态系统|当前只冻结最小状态包；框架允许数据驱动扩展，不硬编码完整官方状态列表|
|护盾系统|保留个人护盾 / team 护盾 / field 护盾这条主动分叉，不伪装成官方原规则|
|信息可见性|标准模式默认采用 `prototype_full_open`；同时预留贴近官方 Open Team List 的模式档位|
|时间控制|原型期使用固定 `max_turn = 40`，不直接照搬官方赛事计时器与抢时规则|

## 4. 统一术语

|术语|定义|
|---|---|
|单位 `unit`|一名可参战角色/宝可梦个体|
|方 `side`|玩家方或敌方|
|在场位 `active_slot`|当前 1v1 场上的唯一位置|
|后备位 `bench`|未在场但仍可上场的单位列表|
|行动 `action`|本回合一次合法进入执行队列的指令|
|行动机会 `action window`|某单位在本回合按排序轮到它执行行动的时刻|
|行动机会已过去 `action_window_passed`|该单位本回合的执行时机已经轮过，不管成功与否|
|成功行动 `has_acted`|该单位的行动已正式开始执行；单纯被睡眠/麻痹/畏缩拦下不算成功行动|
|离场原因 `leave_reason`|`manual_switch / forced_replace / faint`|
|公开信息 `public info`|玩家界面应能看到、AI 也允许读取的信息；当前标准模式下，绝大多数战斗信息都属于公开信息|
|私有信息 `private info`|当前标准模式下仅保留内部随机值、未来未发生的抽样结果、调试字段等运行态信息|
|同一结算窗口|指同一次伤害/效果批次、同一次行动后的击倒检查、同一次回合末结算、或同一次回合上限判定中的同窗比较|

## 5. 全局实现原则

|项|规则|
|---|---|
|排序锁定|一回合的行动队列在“选择指令阶段结束”后锁定，中途不重排|
|系统动作|强制补位不进入本回合行动队列，属于击倒窗口内立即处理的系统动作|
|合法性处理|非法指令必须在选择阶段被拦截，不得带着非法状态进入执行阶段|
|硬非法定义|只拦截硬非法输入；像“满血治疗”“空放净化”这类允许玩家自己选，进执行阶段后正常失败|
|失败语义|要区分“非法、未开始就取消、开始前失败、开始后失效、命中失败、成功结算”|
|离场重置|单位离场时，临时战斗态按统一重置表处理，不能每个技能单独发明口径|
|效果冲突|同触发点统一入同一个效果队列，按统一排序结算|
|同组 field 冲突|按官方天气/场地思路处理：同窗都落地时，后结算者保留|
|系统链路|`battle_init / action / turn_start / turn_end / timeout / system_replace` 等结算段都必须建立 `event_chain_id`|
|资源时序|`turn_start` 的 MP 回复先于 `turn_start` 效果队列；`turn_end` 的 MP 回复后于 `turn_end` 效果队列|
|选指令限时|每方每回合 `30s`；超时未提交合法指令时自动替代默认动作，并写日志|
|绝对先手关系|`absolute_order > 0` 的绝对先手行动优先于手动换人；手动换人仍优先于普通技能|
|异常终止语义|`invalid_battle` 与同窗双方同时投降都按 `no_winner` 结束，不计胜负但必须保留完整日志|
|链深配置|`max_chain_depth` 必须可配置；默认值用于原型期，非硬编码常量|
|日志复现|所有影响结算的随机、排序、离场、触发链都必须可重放|

## 6. 开发里程碑拆分

### 6.1 里程碑 A：最小可玩闭环

|模块|必须落地|
|---|---|
|主流程|战斗开始、完整公开队伍信息、首发、选技、换人、回合推进、击倒补位、结束判定|
|基础伤害|物理 / 特殊 / 真实伤害|
|资源|`MP` 消耗、回合回复、默认动作|
|日志|最小可回放日志：至少记录 `battle_seed / turn_index / action_id / command_type / HP/MP 变化 / 离场 / 胜负`|

### 6.2 里程碑 B：完整战斗规则基线

|模块|必须落地|
|---|---|
|主流程|里程碑 A 全量保留，并补齐完整公开队伍信息下的日志、UI 与 AI 边界|
|基础伤害|物理 / 特殊 / 真实伤害|
|资源|`MP` 消耗、回合回复、默认动作|
|状态|中毒、灼伤、麻痹、睡眠、畏缩|
|持有物|被动持有物触发、揭示、排序与归属|
|效果系统|至少支持 `damage / heal / stat_mod / resource_mod / status_apply / status_remove / rule_mod / apply_global / swap / cleanse`|
|日志|可复现的完整结算日志|

## 7. 首个可玩里程碑不强制实装，但接口允许预留

|内容|口径|
|---|---|
|奥义资源规则|接口保留，入口关闭|
|主动道具模式|只作为未来剧情/PVE 模式扩展，不属于当前标准战斗基线|
|召唤物 / 替身|数据结构可预留，首个可玩里程碑不强制|
|完整属性表|允许后续接入；当前只要求模块化接口与占位|
|更复杂的状态|允许后续扩展，但不得破坏当前状态分组与离场规则|

## 8. 文档变更纪律

|项|规则|
|---|---|
|新增规则|必须写入 `docs/rules/` 对应模块，不得只停留在聊天记录|
|跨模块改动|先改总则，再改模块，再补 `docs/records/decisions.md`|
|历史文件|旧的单文件总表只保留迁移说明，不再承载现行细则|
