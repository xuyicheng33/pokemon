# 模块 02：回合流程、行动排序与失败语义

本文件定义“一回合怎么走、行动怎么排、失败怎么分、何时补位”。

## 1. 回合固定流程

|阶段|规则|
|---|---|
|回合开始|先执行当前在场单位的 MP 回复，再处理 `turn_start` 效果|
|选择指令|双方同时选择本回合指令|
|选择限时|每方 `30s`；超时未提交合法指令则自动替代为 `timeout_default`|
|合法性校验|非法指令立即拦截并重选，不得进入执行阶段|
|队列锁定|双方都提交合法指令后，按统一排序链生成本回合行动队列|
|执行行动|按队列顺序逐个执行|
|行动后击倒窗口|每个行动结算后，立即检查击倒并处理强制补位|
|回合末结算|处理 `turn_end` 效果与 field 剩余回合扣减|
|回合末击倒窗口|回合末结算后再次检查击倒并处理补位|
|胜负判定|一方无可上场单位则判负；同窗双方都无可上场单位则判平|
|回合上限判定|第 40 回合完整结束后，执行超时比较|

## 2. 指令类型

|指令|是否进入行动队列|说明|
|---|---|---|
|技能|是|常规主动技能|
|手动换人|是|trainer 行为，占用本回合行动|
|奥义|是|必须显式声明绝对先手或绝对后手|
|资源型默认动作 `resource_forced_default`|是|当没有任何合法主动方案时自动替代|
|超时型默认动作 `timeout_default`|是|选择超时后自动替代|
|投降|否|即时结束|
|强制补位|否|系统动作，在击倒窗口内立即处理|

补充规则：

1. 被动持有物和 field 不是行动，不进本回合行动队列。
2. 只有会进行动队列的对象，才使用“行动排序”。

## 3. 统一 `priority` 体系

### 3.1 统一语义

|项|规则|
|---|---|
|字段名|统一使用 `priority`|
|数值方向|数值越大越先|
|默认值|未声明时一律为 `0`|
|全局范围|`-5 ~ +5`|

### 3.2 行动侧取值约束

|行动类型|`priority` 规则|
|---|---|
|绝对先手奥义|固定 `+5`|
|手动换人|固定 `+4`|
|普通技能|允许 `-2 ~ +2`，默认 `0`|
|资源型默认动作|固定 `0`|
|超时型默认动作|固定 `0`|
|绝对后手奥义|固定 `-5`|

补充规则：

1. 奥义不能使用 `+4`、`0`、`+2` 这类中间值；它要么绝对先手，要么绝对后手。
2. 普通技能不能写 `+5`、`+4`、`-5` 这种保留值。
3. 标准模式没有“使用持有物”指令，因此不存在主动道具行动优先级。

### 3.3 排序链

行动队列统一使用：

`priority -> speed -> random`

说明：

1. `priority` 相同才看速度。
2. 速度也相同才抽 `speed_tie_roll`。
3. 队列一旦锁定，本回合中途不重排。

## 4. 目标锁定

|目标类型|规则|
|---|---|
|敌方单体|锁定敌方在场位置，不锁定旧单位实例|
|自身|锁定行动者自己|
|field|直接作用在全场 field 作用域上|
|执行时位置已有新单位|命中新单位|
|执行时目标位置为空|若行动已开始执行，则本次行动失效并消耗行动|

## 5. 行动状态术语

|术语|定义|
|---|---|
|`queued`|已通过合法性校验并进入本回合行动队列|
|`action_window_passed`|该单位本回合的行动时机已经轮过|
|`has_acted`|行动已正式开始执行；若之后 miss 或目标无效，仍算 `true`|
|`cancelled_pre_start`|还没轮到执行时，行动者已离场或倒下，队列项被取消|

## 6. 行动失败分类

|类别|发生时点|例子|是否扣 MP|是否消耗本次行动|日志口径|
|---|---|---|---|---|---|
|硬非法指令|选择阶段|MP 不足、没有可换单位、奥义入口配置不合法|否|否|拦截重选，不写执行日志|
|资源型默认动作替代|选择阶段结束前|没有任何合法主动方案|按默认动作规则|是|按正常行动写日志|
|超时型默认动作替代|选择超时|超时前未提交合法指令|按默认动作规则|是|写 `command_type = timeout_default`、`command_source = timeout_auto`|
|未开始就取消|行动轮到前|行动者先被击倒或被强制换下|否|否|记 `cancelled_pre_start`|
|开始前被效果拦下|执行起点|被某个技能定义的持续效果阻断|是|是|记 `action_failed_pre_start`|
|命中失败|命中判定|技能未命中|是|是|触发 `on_miss`|
|开始后失效|已经开始执行后|目标位置为空、条件不满足|是|是|记 `action_failed_post_start`|
|成功结算|正常命中或正常生效|伤害、回复、换人、展开 field|是|是|按完整链路写日志|

## 7. 击倒窗口与强制补位

### 7.1 行动后击倒窗口

|步骤|规则|
|---|---|
|1|单个行动完成后，立即检查双方是否有单位 HP 归 0|
|2|若有倒下，先处理本窗口内全部 `on_faint`、离场清理与相关效果|
|3|若一方需要补位，则该方立即执行强制补位；若双方都需要补位，则双方替补先同时进入 active|
|4|active 状态稳定后，再统一处理新入场单位的 `on_enter` 与相关效果|
|5|若补位引发新的击倒，继续在当前链路中处理到窗口稳定|

### 7.2 回合末击倒窗口

回合末所有效果处理完后，再执行一次与行动后击倒窗口相同的流程。

## 8. 同一结算窗口的定义

|窗口类型|定义|
|---|---|
|行动后窗口|一个行动完整结算后的统一击倒检查窗口|
|回合末窗口|回合末所有效果处理后的统一击倒检查窗口|
|系统效果窗口|同一次系统效果批次造成的统一检查窗口|
|回合上限窗口|第 40 回合完整结束后的超时比较窗口|

同一窗口内若双方都没有可上场单位，则判平局。

## 9. 效果排序入口

同一触发点多个效果同时待处理时，不再另发明第二套词，统一用模块 05 的效果排序链：

`priority -> source_order_speed_snapshot -> source_kind_order -> source_instance_id -> random`

## 10. 胜负与异常终止

|项|规则|
|---|---|
|一方无可上场单位|另一方获胜|
|同窗双方均无可上场单位|平局|
|投降|即时认负|
|`invalid_battle`|立即终止，本场记为 `no_winner`，并保留完整日志|
