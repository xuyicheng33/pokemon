# 模块 02：回合流程、行动排序与失败语义

本文件定义“一回合怎么走、什么时候算出手、失败分几种、何时补位”。

## 1. 回合固定流程

|阶段|规则|
|---|---|
|回合开始|若 `regen_timing = turn_start`，先执行 MP 回复，再处理 `turn_start` 触发|
|选择指令|双方同时选择本回合指令|
|选择限时|每方 `30s`；超时未提交合法指令则自动替代默认动作|
|选择阶段合法性校验|非法指令立即拦截并重选；不得进入执行阶段|
|行动队列锁定|双方指令都合法后，按排序链一次性生成本回合行动队列|
|执行行动|按队列顺序逐个执行|
|行动后击倒窗口|每个行动结算后立即检查击倒，并立刻处理强制补位与入场效果|
|回合末结算|先处理 `turn_end` 触发；若 `regen_timing = turn_end`，再执行 MP 回复|
|持续时间扣减|按效果 `duration_mode` 处理扣减|
|回合末击倒窗口|回合末结算后再次检查击倒，并立刻处理补位|
|胜负判定|若一方无可上场单位，则判负；若同窗双方都无可上场单位，则判平|
|回合上限判定|第 40 回合完整结束后，执行超时比较|

## 2. 指令类型

|指令|是否占用行动|默认 `action_priority`|说明|
|---|---|---|---|
|技能|是|`0`|常规主动技能|
|手动换人|是|`+1`|trainer 行为，占行动；固定 `absolute_order = 0`|
|奥义|是|`0`|接口保留，入口关闭|
|默认动作|是|`0`|当无其他合法主动方案时自动替代|
|强制补位|否|不进队列|击倒窗口内立即执行的系统动作|

## 3. 行动排序

### 3.1 排序链

统一排序链：

`absolute_order -> action_priority -> skill_priority -> speed -> random`

### 3.2 各层定义

|层级|规则|
|---|---|
|`absolute_order`|绝对先手/绝对后手层；当前除显式声明绝对先手/后手的技能或奥义外，默认 `0`|
|`action_priority`|手动换人默认 `+1`；普通技能与默认动作默认 `0`；但不覆盖 `absolute_order > 0` 的绝对先手行动|
|`skill_priority`|仅技能 / 奥义使用，默认 `0`，可由技能定义调整|
|`speed`|读取队列锁定时的有效速度|
|`random`|前四层完全相同才抽随机决胜|

补充规则：

- 手动换人的排序关系固定为：低于 `absolute_order > 0` 的绝对先手行动，高于普通技能、默认动作与非绝对先手奥义。

### 3.3 队列锁定规则

|项|规则|
|---|---|
|锁定时点|选择阶段结束、双方合法指令都确认后|
|中途是否重排|否。回合内即使速度变化、地形变化、麻痹变化，也不重排已锁定队列|
|官方参考口径|对齐官方“速度决定顺序、顺序按回合起始时点确定”的思路；本项目明确写死为不可中途重排|

## 4. 目标锁定

|目标类型|规则|
|---|---|
|敌方单体|锁定敌方在场位置，不锁定旧单位实例|
|自身|锁定当前行动者自身|
|field / team|不锁定单位，直接作用在作用域上|
|执行时位置已有新单位|命中新单位|
|执行时目标位置为空|若该行动已经开始执行，则本次行动失效并消耗行动|

## 5. 行动状态术语

|术语|定义|
|---|---|
|`queued`|已通过合法性校验并进入本回合执行队列|
|`action_window_passed`|该单位本回合的行动时机已经轮过|
|`has_acted`|行动已越过前置阻断检查并正式开始执行；若之后命中失败或目标为空，仍算 `has_acted = true`|
|`cancelled_pre_start`|行动还没轮到时，行动者已倒下或离场，队列项被取消|

补充说明：

- “本回合尚未行动”判断默认看 `action_window_passed == false`。
- “本回合没有行动成”默认看 `has_acted == false`。
- 畏缩的施加条件看 `action_window_passed == false`，不是看 `has_acted`。

## 6. 行动失败分类

|类别|发生时点|例子|是否扣资源|是否消耗本次行动|`action_window_passed`|`has_acted`|触发口径|
|---|---|---|---|---|---|---|---|
|硬非法指令|选择阶段|MP 不足、目标位置不存在、入口关闭、标准模式下选择主动道具（未来模式预留）|否|否|否|否|立即重选，不记入执行日志|
|自动替代默认动作|选择阶段结束前|没有任何合法主动技能，且无合法换人|按默认动作规则|是|后续按默认动作处理|后续按默认动作处理|作为正常行动入队|
|未开始就取消|行动轮到前|行动者先被击倒或被强制离场|否|否|否|否|记录 `cancelled_pre_start`|
|开始前被状态拦下|执行起点|睡眠、麻痹、畏缩阻止技能/奥义|技能/奥义扣资源；换人默认不受这些状态阻断|是|是|否|不触发 `before_action` / `after_action`，记录 `action_failed_pre_start`|
|开始后失效|已经开始执行后|目标位置为空、后续条件不满足|是|是|是|是|触发 `before_action`，再按具体失败分支记日志|
|命中失败|命中判定|普通 miss / 被闪避|是|是|是|是|触发 `before_action -> on_miss -> after_action`|
|成功结算|正常命中或正常生效|伤害、回复、上状态、成功换人|是|是|是|是|按完整事件链处理|

补充规则：

- “满血治疗”“目标当前没有可解状态”这类不属于硬非法，允许进入执行阶段后按失败分支处理。
- 行动者若在自己行动轮到前倒下，则该行动直接取消，不扣资源、不算出手。

## 7. 击倒窗口与强制补位

### 7.1 行动后击倒窗口

|步骤|规则|
|---|---|
|1|单个行动完成后，立即检查双方是否有单位 HP 归 0|
|2|若有倒下，先处理本窗内全部 `on_faint` / `on_kill` / 离场清理|
|3|若只有一方需要补位，则该方立刻执行强制补位并让新单位进入 active；若双方都需要补位，则双方替补先同时进入 active 状态|
|4|active 状态稳定后，再统一处理本窗新入场单位的 `on_enter` 与入场衍生效果队列|
|5|补位引发的新击倒，仍在当前链路中继续处理，直到窗口稳定|

### 7.2 回合末击倒窗口

回合末结算完成后，再执行一次与“行动后击倒窗口”相同的检查与补位流程。

### 7.3 强制补位规则

|项|规则|
|---|---|
|是否延后|否，只要进入击倒窗口且该方仍有后备单位，就必须立刻补位|
|是否进入行动队列|否|
|是否能被取消|只有在该方已经没有可上场单位时才不执行|

## 8. 同一结算窗口的定义

为了避免“双倒下到底算不算同窗”的口头争议，当前把结算窗口固定为以下四类：

|窗口类型|定义|
|---|---|
|单次效果窗口|一次 payload 或一次多段中的当前段，造成的 HP/状态变化批次|
|行动后窗口|一个行动全部结算完成后的统一击倒检查窗口|
|回合末窗口|回合末所有 `turn_end` 与 DOT 结算完成后的统一击倒检查窗口|
|回合上限窗口|第 40 回合完整结束后的超时比较窗口|

同一窗口内若双方都没有可上场单位，则判平局。

## 9. 同速与同触发点排序

|场景|规则|
|---|---|
|两个行动同速同优先级|抽 `speed_tie_roll` 决定先后|
|同一触发点多个效果同时待处理|按 `effect.priority -> source_speed -> source_kind_order -> source_instance_id -> random` 结算；细则见模块 05|
|回合开始/回合末的同速效果|同样走统一效果队列，不额外发明另一套排序规则|

## 10. 胜负与超时判定

### 10.1 常规胜负

|项|规则|
|---|---|
|一方无可上场单位|另一方获胜|
|同窗双方均无可上场单位|平局|
|同窗双方都投降|`no_winner`|
|结算链进入 `invalid_battle`|`no_winner`，且立即终止本场|

### 10.2 回合上限

|项|规则|
|---|---|
|触发时点|第 40 回合完整结束后|
|比较顺序|`alive_count` -> `team_hp_ratio` -> `team_hp_total` -> 平局|

补充公式：

- `alive_count = count(unit.current_hp > 0)`，统计整队 3 名单位中仍存活的数量。
- `team_hp_ratio = sum(unit.current_hp) / sum(unit.max_hp)`，统计整队总剩余 HP 与整队总最大 HP 的比值。
- `team_hp_total = sum(unit.current_hp)`，统计整队总剩余 HP。
- 上述三个比较都把已倒下单位按 `current_hp = 0` 计入。

精度规则（写死）：

1. `alive_count` 与 `team_hp_total` 都按整数比较。
2. `team_hp_ratio` 不做浮点比较，改为交叉相乘比较：比较 `sum_hp_A * sum_max_hp_B` 与 `sum_hp_B * sum_max_hp_A`。
3. 仅当交叉相乘结果完全相等时，才进入下一比较项。

## 11. 选择阶段的输入约束

|项|规则|
|---|---|
|玩家非法输入|必须立即报错并重选|
|AI 非法输入|必须继续搜索直到产出合法指令，不允许把非法指令送进执行器|
|进入执行阶段后|不允许重选|

补充规则：

- 硬非法只包括会破坏执行器前提的输入，如 MP 不足、目标或替补不存在、奥义入口关闭、标准模式未启用的指令类型。
- 满血治疗、空放净化、对当前无可驱散目标使用驱散等，不算硬非法；若玩家坚持选择，按正常队列执行，并在结算时失败。

## 12. 选指令超时处理

|场景|规则|
|---|---|
|`30s` 内提交合法指令|按玩家/AI 提交结果入队|
|`30s` 内未提交合法指令|立刻替代为默认动作，并写入 `timeout_auto_action` 标记|
|已在倒下窗口等待强制补位|不走此限时规则，仍按强制补位系统动作立即处理|
